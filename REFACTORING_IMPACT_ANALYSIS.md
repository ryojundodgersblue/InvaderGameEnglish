# リファクタリングによる機能制限の影響分析

## エグゼクティブサマリー

**結論: 調査で発見された10の根本原因をすべて修正しても、機能の制限は発生しません。**

すべての既存機能（40以上）は維持され、むしろより安定して動作するようになります。
ただし、開発工数と一時的な制約は発生します。

---

## 1. 各根本原因の修正による機能への影響

### 🔴 根本原因 #1: 二重状態管理（React State + 15 Refs）

**修正内容:**
- 15個のRefを削除
- すべての状態をRedux/Context/Zustandなどの単一ソースで管理
- useEffectでの同期処理を削除

**影響を受ける機能:**
- ✅ 音声認識の自動再開（現在: onend内でstatusRef.currentを参照）
- ✅ タイマーのコールバック（現在: deadlineRef.currentを参照）
- ✅ 評価処理（現在: questionsRef.current, idxRef.currentを参照）
- ✅ オーディオのミュート制御（現在: micActiveRef.currentを参照）

**機能が制限されるか？**
❌ **制限されない** - すべての機能は保持される

**実装方法の変更:**
```typescript
// 修正前（Ref使用）
const onend = () => {
  if (statusRef.current === 'listening') {
    recognition.start();
  }
};

// 修正後（Reduxセレクター使用）
const onend = () => {
  const currentStatus = store.getState().game.status;
  if (currentStatus === 'listening') {
    recognition.start();
  }
};
```

**メリット:**
- 状態の同期ズレが完全に解消
- デバッグが容易（Redux DevToolsで状態遷移を可視化）
- 競合状態が発生しない

**開発工数:** 中（3-5日）

### 🔴 根本原因 #3: タイマー競合（setInterval vs 非同期評価）

**影響を受ける機能:**
- ✅ 30秒タイマー表示（現在: 120msごとに更新）
- ✅ タイムアウト検出（現在: remainMs <= 0で判定）
- ✅ タイムアウト処理（現在: handleTimeout呼び出し）

**機能が制限されるか？**
❌ **制限されない** - タイマーは同じ精度で動作

**実装方法の変更:**
```typescript
// 修正前（競合あり）
timerInterval = setInterval(() => {
  if (remainMs <= 0) {
    clearTimer();
    handleTimeout(); // ← evaluateCaptured()と競合可能
  }

// 修正後（状態マシンで排他）

  if (remainMs <= 0) {
    clearTimer();
    stateMachine.send('TIMEOUT'); // ← 処理中なら無視される

```

**メリット:**
- タイマーとの競合がなくなる

**開発工数:** 低（1-2日）

### 🟠 根本原因 #5: オーディオメモリリーク（Promise.race）

**修正内容:**
- Promise.raceパターンを削除
- AbortControllerとAbortSignalを使用
- タイマーを適切にクリーンアップ

**影響を受ける機能:**
- ✅ TTS音声再生（現在: Promise.raceで15秒タイムアウト）
- ✅ オーディオのタイムアウト処理（現在: 15秒で強制停止）
- ✅ 音声終了イベント（現在: audio.onended）

**機能が制限されるか？**
❌ **制限されない** - タイムアウト機能は維持される

**実装方法の変更:**
```typescript
// 修正前（タイマーリーク）
await Promise.race([
  new Promise(resolve => {
    audio.onended = resolve;
    audio.play();
  }),
  new Promise(resolve => {
    setTimeout(resolve, 15000); // ← リークする
  })
]);

// 修正後（AbortSignal使用）
const abortController = new AbortController();
const timeoutId = setTimeout(() => {
  abortController.abort();
  audio.pause();
}, 15000);

try {
  await new Promise((resolve, reject) => {
    abortController.signal.addEventListener('abort', reject);
    audio.onended = resolve;
    audio.play();
  });
} finally {
  clearTimeout(timeoutId); // ← 確実にクリア
}
```

**メリット:**
- メモリリークが完全に解消
- ゲーム1回で48個のタイマーリークが0に
- 長時間プレイでも安定動作

**開発工数:** 中（3-4日）

## 4. 機能制限なしで修正する方法

### 4.1 基本方針

**「実装を変える、機能は変えない」**

すべての修正は、ユーザーから見た動作を完全に保持します。
変わるのは内部のコード構造とアーキテクチャのみです。

### 4.2 具体例: 音声認識の自動再開

**現在の動作:**
1. ユーザーがマイクをONにする
2. 音声認識が自動的に再開する
3. タイムアウト時は自動停止する
4. 問題中は自動的に再起動する

**修正後も同じ動作:**
1. ユーザーがマイクをONにする（変わらない）
2. 音声認識が自動的に再開する（変わらない）
3. タイムアウト時は自動停止する（変わらない）
4. 問題中は自動的に再起動する（変わらない）

**変わるのは内部実装だけ:**
```typescript
// 修正前: Refを使用
if (statusRef.current === 'listening') {
  recognition.start();
}

// 修正後: Reduxを使用
const status = useSelector(state => state.game.status);
if (status === 'listening') {
  recognition.start();
}
```

### 4.3 テスト戦略

**ユーザーシナリオテスト（E2Eテスト）:**
1. ログイン → ステージ選択 → ゲーム開始
2. マイクON → 音声で回答 → 正解
3. マイクOFF → タイムアウト → 次の問題
4. 10問正解 → クリア → ランキング表示

**修正前と修正後で同じ結果になることを確認**
